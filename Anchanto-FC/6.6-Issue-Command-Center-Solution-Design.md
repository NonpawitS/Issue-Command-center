## 0. The one assumption worth challenging first

You asked for a dashboard that **connects directly to the SharePoint Excel files** and is **a quick win with minimal complexity**. Those two goals fight each other.

A true live connection to a SharePoint workbook needs one of: Power Automate flows, the Microsoft Graph API with app registration and OAuth, or migrating the data into a SharePoint List. All of them require IT involvement, credentials, and a hosting story — that is **weeks, not days**, and it is exactly the "complex backend / heavy infrastructure" you said to avoid.

**Recommendation — a staged solution, not a single big build:**

| | Phase 1 — *this week* | Phase 2 — *when it proves useful* |
|---|---|---|
| **Data in** | Drag-drop the Excel export into a browser | Auto-pull from SharePoint (Graph API or scheduled CSV publish) |
| **Backend** | None. One HTML file. | Light: a scheduled job or Power Automate flow that drops a CSV |
| **Effort** | Done (delivered with this proposal) | 2–4 days with IT |
| **Risk** | Zero infra, zero auth, zero approval needed | Needs IT + app registration |

The prototype is built so Phase 2 is a **swap of the data-loading function, not a rewrite**. You get value Monday morning and earn the right to invest in automation later. The rest of this document designs both phases.

## 1. Recommended architecture

### Phase 1 — Zero-backend (delivered)

```
   Operations exports Excel          Single HTML file (browser)               Consumers
   ┌──────────────────────┐         ┌──────────────────────────────┐        ┌──────────────┐
   │ Issuse 6.6 Anchanto  │  drag   │  SheetJS   → parse .xlsx      │        │ Operations   │
   │ Issuse 6.6 Appareal  │ ──────► │  Mapping   → columns→fields   │ ─────► │ IT           │
   └──────────────────────┘         │  Classify  → rules engine     │        │ Management   │
                                     │  Render    → Chart.js + tables│        └──────────────┘
   Config (rules, mapping)           └──────────────────────────────┘
   stored in browser localStorage          no server · no upload
```

**Why this shape:** the entire app is one `.html` file. It can live in a SharePoint document library, a shared drive, or be emailed. Double-click → it opens → drop the Excel → dashboard. Nothing leaves the machine, which also sidesteps the data-privacy question of uploading internal incident data to a cloud service.

### Phase 2 — Light automation (future)

```
   SharePoint workbook ──► Power Automate (scheduled, e.g. hourly)
                              │  exports sheet → CSV
                              ▼
                        SharePoint /published/issues.csv  ◄── dashboard fetches on load
```

The dashboard's `loadData()` simply calls `fetch('issues.csv')` instead of waiting for a drop. Same UI, same logic. No database.

### Data flow (Phase 1)

1. **Ingest** — SheetJS reads every sheet of every dropped file in-browser.
2. **Tag source** — each sheet is labelled `Anchanto-FC` or `FC Ops` (auto-guessed from filename, user-confirmable).
3. **Map** — spreadsheet headers are auto-matched to 10 canonical fields; user overrides once.
4. **Enrich** — each row gets: auto-category, canonical status, normalized severity, vendor-related flag, age in days, recurring-pattern signature.
5. **Render** — KPIs, charts, tables, vendor board, recurring clusters.
6. **Persist** — only the *config* (rules + mapping) is saved to localStorage; the data is re-dropped each session (keeps sensitive data out of storage by default).

### Technology stack — deliberately tiny

| Layer | Choice | Why |
|---|---|---|
| Runtime | Plain browser, **one HTML file** | No build, no install, no server |
| Excel parsing | **SheetJS (xlsx)** via CDN | Reads .xlsx/.xls/.csv client-side |
| Charts | **Chart.js** via CDN | Lightweight, no licence |
| Logic | Vanilla JS | No framework to maintain |
| Storage | `localStorage` | Config persistence, no DB |
| Hosting | File share / SharePoint doc lib | Zero infra |

No React, no Node server, no database, no cloud account. That is the point.

## 2. Data model

A single normalized **Issue** record. Spreadsheet columns are mapped to these 10 canonical fields; everything else is derived.

| Field | Source | Notes |
|---|---|---|
| `id` | mapped | Issue/ticket number; falls back to row index |
| `title` | mapped | Description / problem text — the core signal for classification |
| `date` | mapped | Reported date; **parsed day-first** (dd/mm/yyyy — TH/UK convention) |
| `category` (raw) | mapped | Optional; used if present, else inferred |
| `subsystem` | mapped | OMS/WMS/module — feeds classification |
| `vendor` | mapped | Drives vendor-related flag and tracker |
| `severity` (raw) | mapped | Normalized to Critical/High/Medium/Low |
| `status` (raw) | mapped | Normalized to the 7-stage pipeline |
| `owner` | mapped | PIC / assignee |
| `rootcause` | mapped | Optional; powers the root-cause leaderboard |
| **Derived →** | | |
| `source` | tagged | `Anchanto` or `FC` — the primary cut |
| `categoryAuto` | rules | Business category (see §4) |
| `statusCanonical` | rules | New → … → Closed / On Hold |
| `severityCanonical` | rules | Critical/High/Medium/Low |
| `isVendor` | derived | Vendor named, *or* (no vendor column mapped *and* category is vendor-owned) |
| `ageDays` | computed | Today − reported date |
| `isOpen` | derived | status ≠ Closed |
| `sig` | computed | Category + top keywords → recurring-cluster key |

**Design choice that matters:** the model is *schema-flexible*. The two source files almost certainly have different column names. Rather than forcing one template, the dashboard maps each file's columns to the canonical model. Operations never has to reformat their spreadsheets.

## 3. Issue classification framework

**Key insight:** classification must be a **maintainable rule table owned by Operations**, not hard-coded logic only IT can change.

**How it works:** an ordered list of `category → keywords` rules. For each issue, the engine concatenates *title + raw category + sub-system + root cause*, lowercases it, and assigns the **first category whose any keyword matches**. No match → `Uncategorized` (a useful signal that rules need tuning).

**Default categories (editable in the Settings panel):**

| Category | Example keywords |
|---|---|
| Integration / Interface | integration, api, sync, mapping, oms-wms, sap, idoc, middleware |
| OMS / Order | oms, order, allocation, split, cancel, sla, release |
| WMS / Warehouse | wms, bin, putaway, grn, inbound, outbound |
| Inventory / Stock | inventory, stock, qty, mismatch, count, reservation |
| Picking / Packing | pick, pack, wave, batch, carton, tote |
| Shipping / Carrier | ship, carrier, tracking, manifest, label, awb, dispatch |
| Master Data | sku, product, barcode, uom, catalog, attribute |
| Performance / System | slow, timeout, crash, down, latency, freeze |
| UI / Usability | screen, button, display, layout, ux |
| Configuration | config, setting, parameter, workflow setup |
| Process / Manual | process, manual, training, sop, user error |
| Hardware / Network | printer, scanner, device, network, rf, wifi |

**Why this beats AI/ML classification here:** with ~hundreds of issues and Ops-specific jargon, a transparent keyword table is more accurate, instantly auditable ("why is this 'Integration'?"), and editable by a non-coder in seconds. **What to do next:** after the first real load, review the `Uncategorized` bucket and the Settings panel together for 15 minutes to tune keywords. Export the config and commit it to `config/icc-config.json` so the whole team shares one ruleset.

## 4. Vendor tracking framework

A lightweight 7-stage pipeline — no separate tool, no per-vendor spreadsheet.

```
New → Investigating → Fix In Progress → UAT → Deployed → Closed
                                  └────────→ On Hold (blocked / waiting vendor)
```

The **Vendor Tracker** tab renders this as a board (columns = stages), filterable by vendor, showing each issue's severity and age. Status text in the source file ("pending vendor", "in UAT", "resolved"…) is auto-mapped to a canonical stage via an editable keyword table, so vendors and Ops can keep writing status however they like.

What it answers at a glance:
- **Vendor updates / progress** — where each issue sits in the pipeline, per vendor.
- **Investigation** — count in *Investigating* + age (ageing = stalled).
- **Fix deployment** — *Fix In Progress* → *Deployed* counts.
- **UAT** — the *UAT* column is the pre-release gate.
- **Closure** — *Closed* count and closure rate.

**What to do next:** the single most valuable add to your source spreadsheet is a clean **Status** column using consistent words and a **Vendor** column. The richer those two fields, the sharper this board. Everything else the dashboard infers.

## 5. Executive reporting

The **Executive Overview** answers your six management questions directly:

| Management question | Where it's answered |
|---|---|
| How many issues occurred? | KPI: *Total Issues* (+ Anchanto-FC vs FC Ops split chart) |
| Major issue categories? | *Issues by Category* bar chart (click → drill into the list) |
| % vendor-related? | KPI: *Vendor-Related* with % of total |
| Operational impact? | *Severity Mix* + *Critical Open* KPI + open-issue age |
| What remains unresolved? | KPIs *Open* / *Critical Open*; *Status Pipeline* chart |
| Actions required? | *Recurring Patterns* count + *Top Root Causes* (where to fix once, not many times) |

KPI cards: **Total · Open · Closed · Vendor-Related % · Critical Open · Recurring Patterns · Avg Open Age**. A scope toggle in the header (**All / Anchanto-FC / FC Ops**) re-cuts the *entire* dashboard — the explicit separation you asked for. Charts are click-to-drill: click a category bar → jump to that filtered issue list.

## 7. HTML implementation strategy (how it's built)

- **One self-contained file**, libraries from CDN — copy it anywhere, it works offline after first load.
- **Pure functions** for the brains (`classify`, `normStatus`, `parseDate`, `signature`, `clusters`) — independently testable; all 13 unit tests pass, including day-first date handling.
- **Config-driven**, not code-driven — categories, status mapping, column mapping, source tags all live in an editable Settings panel and an exportable JSON.
- **Defensive parsing** — tolerates messy real-world data: blank cells, mixed date formats, Thai or English headers, multiple sheets, inconsistent status wording.
- **Phase-2-ready** — data loading is isolated in one function; swapping drag-drop for a SharePoint `fetch()` doesn't touch the UI or logic.

## 8. SharePoint connection plan (your Phase-2 ask)

Three options, lightest first:

1. **Power Automate → published CSV (recommended).** A scheduled flow exports the workbook sheet to `issues.csv` in a doc library. Dashboard fetches it on load. No app registration, no code, Ops-grade. ~½ day with someone who has Power Automate access.
2. **Microsoft Graph API.** Dashboard reads the workbook live via Graph. Needs Azure app registration + OAuth + admin consent. More "live", much more setup and an IT/security review. 2–4 days.
3. **Migrate to a SharePoint List.** Strongest long-term (real fields, validation, history) but it changes how Ops *enters* issues. A project, not a quick win.

**My call:** ship Phase 1 now, pilot for 2–3 weeks, and only then do **Option 1** if the manual drop becomes a chore. Don't pay the Graph/List cost until the tool has proven it earns its keep.

## 9. Deployment recommendation (you asked me to advise)

**Single HTML file on SharePoint / shared drive — no hosting.** Reasons: Ops, IT, and Management all already have file access; there's nothing to deploy, secure, or pay for; sensitive incident data never leaves the user's browser. If you later want a clean internal URL, the same file can be dropped onto any static host (SharePoint page, internal IIS, or an Azure Static Web App) with zero code changes — but that's optional polish, not a requirement.

## How to use it today (2-minute quick start)

1. Open **`Issue-Command-Center.html`** (double-click — any modern browser).
2. Click **▣ Sample data** to see it fully populated immediately, *or* drag your two real Excel files onto the page.
3. Open **⚙ Settings** → confirm each file is tagged Anchanto-FC vs FC Ops, check the column mapping, tune category keywords. Click **Apply**.
4. Use the header **All / Anchanto-FC / FC Ops** toggle to cut the whole dashboard.
5. **Export config** and save it to `config/` so your team shares one ruleset.

> Built to be honest about trade-offs: it does the 80% that delivers value this week, and it's structured so the remaining 20% (live SharePoint sync) is an upgrade, not a rebuild.
